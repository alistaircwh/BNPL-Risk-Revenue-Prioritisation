## Project Overview

Buy Now Pay Later (BNPL) industry data science project. The goal is to identify which merchants BNPL should prioritise by building fraud detection models, revenue forecasting (LSTM), and a weighted merchant ranking system.

Stack: PySpark (ETL), scikit-learn (fraud models), TensorFlow/Keras (LSTM ranking), pandas, geopandas, folium, matplotlib/seaborn.

## Running the Project

There is no build system. All work is in Jupyter notebooks that **must be run sequentially** due to data dependencies:

```
notebooks/1_ETL_pipeline.ipynb          # Produces data/curated/ and data/tables/
notebooks/2.1_preliminary_analysis.ipynb
notebooks/2.2_geospatial_analysis.ipynb
notebooks/2.3_visualisation.ipynb
notebooks/3.1a_consumer_fraud_model_v1.ipynb
notebooks/3.1b_consumer_fraud_model_v2.ipynb
notebooks/3.2_merchant_fraud_model.ipynb
notebooks/4.1b_ranking_model_v2.ipynb
notebooks/4.2_segments.ipynb
notebooks/5_summary.ipynb               # Summary/review of all findings
```

Notebook 1 must run first — it generates `data/curated/` and `data/tables/` which are gitignored and not present in the repo.

## Scripts Architecture

`scripts/` contains reusable Python modules imported by notebooks — they are not standalone executables.

| Script | Role |
|--------|------|
| `etl_pipeline.py` | Core ETL: merchant/consumer cleaning, datetime filtering, KNN imputation for LGA codes and income metrics, ABS data join |
| `consumer_transaction_model.py` | Consumer fraud ML pipeline: log transforms, normalisation, Decision Tree / Random Forest / Linear Regression training utilities |
| `merchant_fraud.py` | Merchant fraud model: feature normalisation, CrossValidator hyperparameter tuning, RMSE/R² evaluation |
| `ranking_model_v2.py` | LSTM revenue forecasting (3-period ahead); weight generation for merchant ranking |
| `consumer_model.py` | Consumer-level visualisation utilities |
| `preliminary_analysis.py` | Dataset profiling: row counts, missing/NaN values |
| `geospatial_analysis.py` | Folium choropleth map creation by postcode |
| `visualisation.py` | Merchant revenue pie charts, top-N merchant plots |

## Data Layout

```
data/raw/        # Tracked in git — shapefiles, postcodes CSV, income/fraud xlsx
data/curated/    # Gitignored — generated by notebook 1 (cleaned/joined data)
data/tables/     # Gitignored — generated by notebook 1 (partitioned transaction tables)
```

See [data/README.md](data/README.md) for authoritative schema documentation.

### Key Data Quirks

- **Transactions**: contain 10% random noise; a separate fraud delta file applies fraud labels
- **Merchant tags**: pipe-delimited strings requiring parsing; take rates embedded in tag field
- **Consumer addresses**: fake US-style addresses but postcode field contains real Australian postcodes (used for geo joins)
- **ID mapping**: `user_id` in transactions must be joined to `consumer_id` via the mapping table before joining with consumer data
- **Revenue levels**: merchant revenue encoded as letters `a`–`e`
